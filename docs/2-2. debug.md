## 1. 资源路径映射错误 (404 错误)
- 现象: 浏览器控制台报 index.tsx 和 index.css 404 错误。
- 原因: index.html 中引用的路径是根目录 /index.tsx，而实际代码位于 src/ 目录下。
- 解决:
    - 将 **index.html 中的脚本引用从 /index.tsx 修改为 /src/main.tsx**（后按要求改为 /src/index.tsx）。
    - 移除了不存在的外部 index.css 引用。
## 2. 模块解析与目录导入失败
- 现象: Vite 无法解析 ./types 或 ./components/Modal 等导入路径，导致**白屏**。
- 分析：
    - **导入路径指向的是文件夹时（比如`import ... from './components/Modal'`），构建工具vite会默认寻找该文件夹下的index.xx文件，找不到则报错。**
    - **而导入路径具体指向了文件名时（比如`import AdCard from './components/AdCard/AdCard.tsx'`），那么不用命名为index.xx文件，vite不会报错。**
## 3. Importmap 导致的运行时冲突
- 现象: 应用启动后显示白屏，且没有明显的错误捕捉。
- 原因: index.html 中包含了一个 importmap，将 react 强制指向了 ESM.sh 上的 React 19，而项目本地使用的是 React 18。这种外部映射与 Vite 的本地模块链接机制产生了严重冲突。
- 分析：
    - importmap是一种现代浏览器原生支持的特性，允许你定义模块路径的映射。比如代码里写 import React from 'react'，浏览器会直接去 esm.sh 下载对应的 React，而不是通过 Vite 去 node_modules 找。
    - **React 的 Hook（如 useState）要求必须在同一个 React 实例的上下文中运行**。版本冲突会导致 Invalid hook call 错误，或者渲染器找不到对应的 React 实例，从而导致程序直接由于报错停止运行。由于**报错发生在 root.render() 之前或过程中，页面上什么都不会显示，即“白屏”**。
- 解决: 彻底从 index.html 中删除了 importmap 块，让应用完全回归通过 Vite 管理本地依赖。
- 一般化：对任意依赖，指向版本 1 但安装版本 2 会白屏吗？
    - 普通的工具库（如 lodash），通常不会白屏。但对于具有**破坏性变更 (Breaking Changes) 或单例模式** (Singleton) 的库（如 React, Vue, Styled-components），版本冲突极高概率触发白屏。
    - 依赖冲突会触发白屏：框架核心库比如React, Vue；**样式引擎**：Styled-components, Emotion；路由库：React-Router（多个 Router 实例冲突）；状态管理：Redux, MobX（版本不匹配可能导致状态分发失效）；
## 4. 模块链接阶段的“静默故障” (致盲点)
- 现象: 即使路径正确，应用依然渲染为空白（#root 为空），且控制台没有明显的 JS 报错。
- 原因: 通过浏览器 Agent 进行动态导入测试发现，src/api/geminiService.ts 文件是空的，但它被 Modal 组件强依赖。由于缺少 generateAdCopySuggestion 函数的导出，导致 React 应用在模块**链接阶段挂起**，无法进入**渲染流程**。由于 JavaScript 是单线程执行的，一个顶层的 SyntaxError 会直接终止整个脚本的后续执行。这意味着 
main.tsx中的 **root.render()** 永远不会被调用，页面永远停留在 HTML 初始状态（即 #root 为空），表现出来的就是白屏
    - 强依赖：指 A 模块在顶层（Top-level）通过 import 直接引入了 B 模块。这意味着 B 模块必须在 A 模块执行前成功加载并解析。
- 解决: 为 geminiService.ts 编写了一个模拟 (Mock) 的导出函数，确保模块引用链完整。

## 5. 环境配置与构建错误
- 现象: 执行 pnpm build 时报错找不到 node 的类型定义。
- 原因: **tsconfig.json** 配置了 types: ["node"]，但 package 内部没有安装对应的 @types/node。
- 解决:
  - 安装了 @types/node 开发依赖。
  - 优化了 vite.config.ts 中的 define 配置，防止环境变量未定义时导致混淆。
- tsconfig.json文件：是 TypeScript 的配置文件，定义了如何将 TS 代码编译成 JS。
```json
{
  "compilerOptions": {
    "target": "ES2022",                // 编译出的 JS 符合 ES2022 标准（较现代）
    "module": "ESNext",                // 使用最新的 ESM 模块系统
    "lib": ["ES2022", "DOM", ...],     // 代码中可以调用的库（包含浏览器 API）
    "moduleResolution": "bundler",      // 告诉 TS 像 Vite 这样的打包工具一样解析模块
    "jsx": "react-jsx",               // 使用新的 React 17+ JSX 转换逻辑
    "paths": { "@/*": ["./*"] },       // 路径别名配置，让 @ 指向根目录
    "allowImportingTsExtensions": true, // 允许在 import 中带 .ts 后缀（Vite 常用）
    "noEmit": true                     // 只进行类型检查，不生成 .js 文件（交给 Vite 做）
  }
}
```
## 6. 关于package.json文件
- dependencies (生产依赖)：应用**运行时**必须的包（如 react, react-dom）。
  - 如下所示：业务逻辑组件库（UI 组件、工具类、类型定义）、react, react-dom放在dependencies中。
- devDependencies (开发依赖)：仅在**开发、构建、测试时**需要的包（如 tailwindcss, vite, typescript, @types/react）。它们不会被打包进最终用户运行的 JS 代码中。
  - 如下所示：vite, typescript、eslint、prettier（**@repo/eslint-config、@repo/tsconfig？**）、@types/*（编译时提供类型提示）放在devDependencies中。

## 7. 编辑器中全局查找某个关键词
- IDE方式：`Cmd + Shift + F`
- 命令行：查找比如关键词“Layout”
    ```bash
    grep -r "Layout" .
    # 或者排除 node_modules 以提速
    grep -r "Layout" . --exclude-dir=node_modules
    ```
