```
// 使用 ModalType 统一管理，职责清晰，扩展性极强
const [modal, setModal] = useState<{
    type: ModalType;        // 'DETAIL' | 'FORM' | 'DELETE' | 
    'REJECT_REASON' | null
    ad: Ad | null;          // 当前操作的广告对象
    formMode: 'CREATE' | 'EDIT';
}>({ type: null, ad: null, formMode: 'CREATE' });

// 一个通用函数搞定所有打开操作
const openModal = (type: ModalType, ad: Ad | null = null, 
formMode: 'CREATE' | 'EDIT' = 'CREATE') => {
    setModal({ type, ad, formMode });
};

// 统一的确认处理逻辑
const handleConfirm = (payload?: any) => {
    if (modal.type === 'DELETE') methods.deleteAd(modal.ad.id);
    if (modal.type === 'FORM') {
        modal.formMode === 'CREATE' ? methods.addAd(payload) : 
        methods.updateAd(modal.ad.id, payload);
    }
    closeModal();
};
```
说明 ：

- 低耦合 ：优化后，组件不需要关心具体哪个布尔值为 true ，只需判断 modal.type 。
- 易维护 ：新增弹窗类型时，只需在 ModalType 中增加一个枚举值，无需在 Hook 中增加新的 useState 。
- 状态同步 ：解决了旧版本中可能出现的“多个弹窗同时被设为 true”的潜在 Bug。


### 大厂项目中 StatsCard 数据是如何获取的？
在大厂的生产环境下，统计数据（Stats）通常不会通过前端遍历数组计算，也不会写死。主要有以下三种路径：

- 路径 A：后端聚合接口（最主流）
  - 逻辑 ：前端调用一个专门的 /api/v1/dashboard/stats 接口。
  - 原因 ：当广告数据达到上万条时，前端计算会导致浏览器卡顿；且后端数据库（如 ClickHouse, Elasticsearch）在做聚合查询（Count, Sum, Group By）时效率极高。
- 路径 B：BFF 层聚合（Node.js 层）
  - 逻辑 ：在 BFF (Backend For Frontend) 层同时请求列表数据和统计数据，组装成一个大对象返回给前端。
- 路径 C：全局状态管理（跨页面共享）
  - 逻辑 ：使用 Redux / Zustand / TanStack Query 维护数据。统计数据被存放在全局 Store 中，页面加载时触发 Fetch。
