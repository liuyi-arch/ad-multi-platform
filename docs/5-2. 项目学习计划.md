### 为什么先理清现状？

1.  **建立心智模型：** 直接学习 `Zustand + React Query` 就像是给你一辆跑车，但你还不知道城市里的基本路况。首先理解当前代码（即使它不完美）是如何工作的，能帮助你建立一个关于项目数据流和业务逻辑的**基础心智模型**。你知道了数据从哪里来，到哪里去，解决了什么问题。
2.  **理解“为什么”：** 只有理解了当前方案的**痛点**（例如，`useAdsData` 的手动 `loading` 管理，`useAuth` 的状态无法全局共享），你才能真正深刻地理解 `Zustand` 和 `React Query` **为什么**是更好的方案。在面试中，能够清晰地阐述“我们遇到了什么问题，因此我们选择了某某方案来解决它”远比“我知道某个方案很好”更有说服力。
3.  **降低学习曲线：** 在不熟悉项目的情况下，同时引入两个新的、功能强大的库，会让你面临巨大的学习压力。你不仅要理解新库的 API，还要思考如何将它们应用到一个你本就不熟悉的项目结构中，很容易感到不知所措。
4.  **面试考察深度：** 面试官很可能会问你“项目中某个具体功能是怎么实现的？”如果你只知道用 `React Query` 后的方案，而对之前的实现一无所知，会显得你对项目的理解不够深入。能够对比新旧两种方案的优劣，是展现你技术深度和思考能力的绝佳机会。

#### **阶段一：理解现状，掌握核心业务流 (预计 1-2 天)**

**目标：** 能够独立画出核心模块的数据流图，并口头描述其工作流程。

1.  **深入 Auth 模块 (建立扎实基础)**
    *   **任务：** 彻底理解用户认证的完整生命周期。这不仅仅是看代码，而是要能解释清楚每个环节的职责和交互。
    *   **关键文件：**
        *   **入口页面:** `apps/admin-web/src/pages/AuthPage.tsx`, `apps/advertiser-web/src/pages/auth/index.tsx`
        *   **UI 组件:** `packages/ui-components/src/Auth/*` (AuthLayout, AuthTabs, LoginForm, RegisterForm)
        *   **核心逻辑:** `packages/hooks/src/useAuth.ts`
        *   **API 客户端:** `packages/api/src/authService.ts`
        *   **数据契约:** `packages/shared-types/src/auth.ts`
        *   **后端逻辑:** `apps/server/src/modules/auth/auth.controller.ts`, `auth.model.ts`
        *   **全局集成:** `apps/admin-web/src/App.tsx` (查看 `ToastContainer` 和 `isAuthPage` 逻辑)
    *   **思考问题 (面试高频点):**
        *   `admin-web` 和 `advertiser-web` 的 `AuthPage.tsx` 如何复用相同的 `useAuth` Hook 和 UI 组件，但实现了不同的登录后跳转逻辑？这体现了什么设计思想？
        *   在 `LoginForm.tsx` 中，`{...auth}` 这种 props 展开的写法有什么优缺点？
        *   `useAuth` Hook 是如何通过 `onNotify` 回调与 `Toast` 组件通信的？这是一种什么模式？
        *   为什么选择 `localStorage` 存储 Token？它有什么安全风险（XSS）？你能提出一个更安全的替代方案吗（HttpOnly Cookie）？
        *   后端的 `auth.model.ts` 是如何通过 `toJSON.transform` 防止密码等敏感信息被发送到前端的？
        *   后端的 `auth.controller.ts` 中，密码校验的逻辑 (`user.password !== password`) 存在什么致命问题？你会如何用 `bcrypt` 修复它？
    *   **关联技术点：** React Hooks (`useState`, `useEffect`), `localStorage`, `async/await`, Koa.js, Mongoose Schema & `transform`, RESTful API, JWT (概念), 密码安全 (哈希)。

2.  **理清广告 CRUD 模块**
    *   **任务：** 模仿 Auth 模块的分析方法，理清“获取广告列表”、“创建广告”、“更新广告”这三个核心流程。
    *   **关键文件：** `useAdsData.ts`, `useAdsModal.ts`, `adService.ts`, `apps/server/src/modules/ad/*` (假设有这个模块)。
    *   **思考问题：**
        *   `useAdsData` 是如何获取和更新广告列表的？
        *   `loading` 和 `error` 状态是如何被手动管理的？
        *   `useAdsModal` 是如何与 `useAdsData` 交互来触发数据更新的？
    *   **关联技术点：** `useCallback`, `Promise`, RESTful API 设计。

3.  **理解数据同步模块 (WebSocket)**
    *   **任务：** 理清 `useWebSocket.ts` 的工作原理。
    *   **关键文件：** `useWebSocket.ts`, `apps/server/src/...` (处理 WebSocket 连接的部分)。
    *   **思考问题：**
        *   前端是如何建立 WebSocket 连接的？
        *   当后端推送消息（如 `AD_STATUS_CHANGED`）时，前端是如何接收并更新 `useAdsData` 中的 `ads` 状态的？（提示：很可能是在 `App.tsx` 中通过 `onMessage` 回调来调用 `setAds`）。
    *   **关联技术点：** WebSocket 协议, `useRef`, `useEffect` 清理函数。

4.  **文件上传模块 (如果存在)**
    *   **任务：** 找到处理文件上传的前端组件和后端控制器。
    *   **思考问题：**
        *   前端是如何使用 `FormData` 来构造请求的？
        *   后端是如何处理 `multipart/form-data` 类型的请求的？（通常需要 `koa-multer` 或类似的中间件）。
        *   文件是如何存储的？（本地服务器？云存储如 OSS/S3？）。
    *   **关联技术点：** `FormData`, `multipart/form-data`, 文件存储策略。

**阶段一产出：** 你应该能为每个模块画出类似我们之前画的流程图，并能向面试官清晰地解释每个流程。

#### **阶段二：引入并学习优化方案 (预计 2-3 天)**

**目标：** 掌握 `Zustand` 和 `React Query` 的核心用法，并能用它们重构现有代码，清晰地阐述新旧方案的优劣。

1.  **学习 React Query (服务器状态)**
    *   **任务：** 阅读 React Query 官方文档的核心概念部分。重点理解 `useQuery`, `useMutation`, `QueryClient`, `queryKey`。
    *   **实践：**
        *   创建一个新的 `useAdsQuery.ts` Hook。
        *   用 `useQuery` 替换 `useAdsData` 中的 `fetchAds` 逻辑。
        *   用 `useMutation` 替换 `addAd`, `updateAd`, `deleteAd` 逻辑，并在 `onSuccess` 回调中调用 `queryClient.invalidateQueries(['ads'])` 来自动刷新列表。
    *   **对比总结：** 写下笔记，对比 `useAdsData` 和 `useAdsQuery` 的代码量、健壮性和心智负担。

2.  **学习 Zustand (客户端状态)**
    *   **任务：** 阅读 Zustand 官方文档，理解其基本用法。
    *   **实践：**
        *   创建一个 `authStore.ts`，将 `useAuth` 中的 `user`, `token`, `role` 状态及其更新逻辑迁移进去。
        *   重构 `useAuth`，使其专注于表单 UI 逻辑，并从 `authStore` 获取状态和调用 actions。
        *   创建一个 `ProtectedRoute` 组件，从 `authStore` 读取登录状态，实现真正的路由保护。
    *   **对比总结：** 思考如果用 React Context 来实现全局 Auth 状态，会遇到什么问题（性能、模板代码），Zustand 是如何解决的。

**阶段二产出：** 你不仅掌握了两个强大的库，还拥有了可以展示给面试官的、经过重构的、更高质量的代码，并且能清晰地解释你这样做的原因。

#### **阶段三：扩展与深化 (持续进行)**

**目标：** 拔高你的技术视野，为面试中的开放性问题和系统设计问题做准备。

1.  **深入安全知识：**
    *   **任务：** 学习并实践我们之前讨论过的安全点：`bcrypt` 密码哈希、`jsonwebtoken` (JWT) 的使用、`HttpOnly` Cookie 的原理。

这个修订后的计划将第一阶段的任务具体化了，为你提供了一个更清晰的起点。请从**阶段一**开始，逐个模块地去理解和梳理，当你对现状了然于胸时，再进入阶段二，你会发现学习和应用新技术的效率和深度都会大大提高。
